# stem-dictionaries-manager Constitution

## Core Principles

### I. Spec-Driven Development (NON-NEGOTIABLE)

Every non-trivial feature MUST flow through the speckit pipeline: `/speckit-specify`
→ optional `/speckit-clarify` → `/speckit-plan` → optional `/speckit-checklist` and
`/speckit-analyze` → `/speckit-tasks` → `/speckit-implement`. Specifications describe
**WHAT** and **WHY**; never **HOW** (no stack, APIs, or code structure inside `spec.md`).
Each feature gets its own numbered branch (`feat/###-name` or matching speckit prefix).

Trivial fixes (typos, dependency bumps, single-line bug fixes, mechanical renames) MAY
bypass speckit but MUST still follow workflow discipline (Principle V).

**Rationale**: Specs are the source of truth that survives refactors and the planned
F# migrations (Phases 3–5). Without speckit's `spec.md`, intent has to be re-derived
from code each time a layer is rewritten. Lean formalization (where applicable) plugs
in directly: prose invariants from `spec.md` become Lean predicates; preservation
theorems gate security-critical state machines.

### II. STEM v1 Standards Are the Contract

The eight STEM v1 standards inline at `docs/Standards/` (REPO_STRUCTURE, LANGUAGE,
MODULE_SEPARATION, PORTABILITY, BUILD_CONFIG, TESTING, CI, MIGRATION) are the
load-bearing cross-repo contract. The Standard version pinned in the top-level
`CLAUDE.md` is authoritative.

- `docs/Standards/*.md` MUST NOT be edited directly — they are regenerated by
  `apply-repo-standard.ps1` on bump. Upstream edits live in
  `llm-settings/shared/standards/`.
- When a topic falls under a standard's scope (TFM choice, project layout, banned
  APIs, language-per-layer, central package management, test framework, CI gates,
  migration cadence), the standard MUST be read and followed. This constitution
  references the standards but does not duplicate them.
- Drift discovered in passing is recorded as a separate `chore:` issue, not folded
  into an unrelated PR.

**Rationale**: STEM has a multi-repo .NET 10 fleet (button-panel-tester, device-
manager, production-tracker, communication, dictionaries-manager). Re-deriving
conventions per repo produces drift; one source of truth in
`llm-settings/shared/standards/` keeps all repos coherent at a single maintenance
location.

### III. Test-First, Manual Fakes, Integration over Mocks (NON-NEGOTIABLE)

Tests are written **before** implementation for any feature with non-trivial logic
(state machines, validation, persistence, authentication, authorization, mappers
with branching).

- Mocking libraries (Moq, NSubstitute, FakeItEasy, etc.) are **forbidden**. Manual
  fakes only.
- Integration tests use SQLite in-memory via the existing `IntegrationTestBase` /
  `ApiIntegrationTestBase` (`tests/Tests/Integration/`). Never a separate mock DB.
- One tests project per repo (per TESTING standard) until the conditions to split
  are explicitly met.
- xUnit method naming: `{Method}_{Scenario}_{ExpectedResult}`.
- TODO(LEAN_WORKSPACE): once a `specs/` Lean 4 workspace exists in this repo,
  preservation theorems MUST gate security-critical state machines (Lean spec →
  xUnit test → .NET impl, in that order).

**Rationale**: Mocking libraries optimise for "tests that pass" rather than "tests
that catch bugs". SQLite-in-memory has matched production semantics for our query
patterns across STEM repos and is fast enough for full-scenario seed-and-assert
flows. Manual fakes force a thinking pass on the boundary every time, which surfaces
bad seams.

### IV. Pragmatic .NET — Explicit, Nullable, Exceptional

Hard rules across all C# / F# in this repo:

- `<Nullable>enable</Nullable>` MUST be set on every project. Nullable warnings are
  errors (BUILD_CONFIG enforces `TreatWarningsAsErrors`).
- DI is **manual in the composition root** (`Program.cs` for the API project,
  `Composition/` in the future Avalonia GUI project). No service-locator helpers,
  no auto-registration scanners, no `Scrutor`-style assembly scans.
- Errors propagate via **exceptions**, not `null` returns or `Result<T>` wrappers.
  `null` returns are reserved for "not-found" query semantics on `GetBy*Async`.
- Constructor validation uses `ArgumentNullException.ThrowIfNull` and
  `ArgumentException.ThrowIfNullOrWhiteSpace`.
- Function bodies SHOULD be ≤ 15 LOC. Soft column limit 100–110, hard 120.
  Justify exceptions inline.
- All user-visible artifacts (code identifiers, XML docs, inline comments, GUI
  strings, CHANGELOG, commit bodies, PR descriptions, issue text) are in **English
  by default**. Italian only when explicitly requested per artifact. Phase 2 (#33)
  is closing the residual Italian content.

**Rationale**: STEM is a one-developer shop today; ceremony has no payoff and slows
the F# migration tracks. Manual fakes + manual DI + exceptions is the path of least
resistance for code that ships and stays maintainable through the planned rewrites.

### V. Workflow Discipline (NON-NEGOTIABLE)

- **Never commit on `main` / `master`**. All work flows through feature branches
  (`feat/`, `fix/`, `refactor/`, `docs/`, `chore/`).
- **Conventional commits** with scope where helpful: `feat(api): …`,
  `fix(infra): …`, `refactor(core): …`, `docs(standards): …`, `chore: …`. Bodies
  are factual and low-profile. **No AI attribution** in commits, PRs, or issues.
- **Dual-remote**: `github` is the active remote (PRs, Actions CI, issues,
  Artifacts, project board). `bitbucket` is mirror-only via the
  `.github/workflows/mirror-bitbucket.yml` workflow on push to `main`. Direct
  pushes to `bitbucket` are blocked at the URL level (`set-url --push no_push`).
- **PRs on GitHub only**, never on Bitbucket. **Issues on GitHub only**.
- **CI of record is GitHub Actions**. Both `ubuntu-latest` and `windows-latest`
  matrix legs MUST pass before merge.
- **CHANGELOG.md** updated under `[Unreleased]` with every PR (per the
  `stem-conventions` rule).
- **Rebase merge** preferred (linear history). Squash only when a feature
  branch's commits are inherently noisy.

**Rationale**: STEM's mirror requirement (Bitbucket visibility for non-developer
staff) and GitHub-native productivity (Actions, gh CLI, Projects v2) need both
remotes honest. The Actions mirror is the only path that keeps `bitbucket/main`
in sync after `gh pr merge` server-side merges — local pushurls cannot reach those.

## Security & Auditability

This is an API server that ships dictionary data to external supplier installations.
Security and auditability are first-class:

- **Secrets never live in source or in client-shipped configs** that an attacker
  can extract from a supplier's filesystem. The `appsettings.json` `ApiKeys` section
  is acceptable for trusted internal services on infrastructure we control;
  per-installation credentials for **external supplier apps** MUST be issued via the
  bootstrap-token registration flow (issue #1).
- **Bootstrap tokens** are single-use, time-bounded, and supplier-scoped. Long-lived
  per-installation API keys are stored hashed (PBKDF2 / SHA-256 HMAC, never
  plaintext at rest); the plaintext returned at `/register` time is the only
  opportunity for the client to capture it.
- **Per-installation API keys are independently revocable**. One supplier's
  compromised key MUST NOT blast-radius to other installations.
- **Audit on every state mutation**. Domain CRUD writes `AuditEntry` rows via
  `IAuditService.LogCreateAsync` / `LogUpdateAsync` / `LogDeleteAsync`. Registration
  and credential events MUST be logged with timestamp, supplier, source IP, and
  `installation_descriptor` (storage shape — extend `AuditEntry` vs. dedicated
  `RegistrationLog` table — is decided per feature in its `/speckit-plan`).
- **Failure paths leak nothing**. Authentication / registration errors return `401`
  (or the spec's chosen non-distinguishing code) with a body that does NOT reveal
  whether the token was missing, expired, already used, or scoped to a different
  supplier.
- **Banned APIs** (`BannedSymbols.txt`, see PORTABILITY) is enforced at build time.
  Pure layers (`Core`, `Services`) MUST NOT reference `System.Drawing.*`,
  `Microsoft.Win32.Registry`, `System.Management`, raw `System.IO.Ports.SerialPort`,
  or hardcoded paths.

## Quality Gates

A change is mergeable only when **all** of the following pass:

1. `dotnet format whitespace --verify-no-changes --no-restore` — green (CI's
   whitespace-only gate; the local Husky.NET hook runs the full
   `dotnet format --verify-no-changes`).
2. `dotnet build -c Release` — green with `TreatWarningsAsErrors` (per BUILD_CONFIG).
3. `dotnet test -c Release` — green on both `ubuntu-latest` and `windows-latest`.
4. **Banned APIs** — no introductions of banned APIs in pure layers.
5. **Constitution Check** — `/speckit-plan` output's gate section reflects compliance
   with this constitution; deviations are recorded in the plan's Complexity Tracking
   table with both a justification and a "simpler alternative rejected because" line.
6. **CHANGELOG.md** — new entry under `[Unreleased]` describing user-visible impact.
7. **Standard version unchanged** in this PR — bumping the `docs/Standards/` pin is a
   separate `chore: bump standards to vX.Y.Z` PR per the MIGRATION standard.

## Governance

- **Authority**: This constitution supersedes ad-hoc decisions. When a tactical
  choice conflicts with a principle here, the principle wins; if a principle is
  itself wrong, amend the constitution first via PR (see below) — do not work
  around it.
- **Amendments**: Require a PR. Version bump rules (semver):
  - **MAJOR**: a principle is removed or its semantics flip (e.g., dropping the
    no-mocking-libs rule, or allowing commits on `main`).
  - **MINOR**: a new principle or section is added, or guidance is materially
    expanded.
  - **PATCH**: clarifications, wording, typo fixes, non-semantic refinements.
- **Compliance review**: `/speckit-plan` runs a Constitution Check gate. Any
  violation MUST be either resolved before plan approval or recorded in the plan's
  Complexity Tracking with a written justification and a simpler-alternative-
  rejected-because. Plans with un-justified violations are not approved.
- **Runtime guidance precedence** (most-specific wins):
  1. This constitution (`.specify/memory/constitution.md`).
  2. The repo `CLAUDE.md` (Standard version pin, repo-specific notes, language
     deviations, active migrations).
  3. `docs/Standards/*.md` inline copies (the eight v1 standards).
  4. `~/.claude/CLAUDE.md` and `claude/rules/*.md` (global rules: communication,
     dual-remote, no-attribution, promote-to-llm-settings, stem-conventions).
  5. Skills under `~/.claude/skills/` (workflow, dotnet, github-actions, lean4,
     pr, documentation, …).
